# Git Workflow and Pull Request Rules

## Ветки
- `main` — production-ветка, прямой пуш запрещён.
- `develop` — ветка разработки.
- Feature-ветки: `feature/короткое-описание`.
- Bugfix-ветки: `fix/короткое-описание`.

## Коммиты
- Используйте Conventional Commits: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`.
- Сообщение коммита — на русском или английском.
- Один коммит — одно логическое изменение.

## Pull Request
- Название PR должно отражать суть изменений.
- Описание PR должно содержать:
  - Что сделано.
  - Почему сделано именно так.
  - Как тестировать.
  - Скриншоты (для UI-изменений).
- PR должен быть привязан к задаче в трекере.

## Ревью
- Минимум 1 одобрение от члена команды перед мержем.
- Все замечания ревьюера должны быть исправлены или обоснованно отклонены.
- Ревьюер проверяет: код-стайл, тесты, доступность, безопасность.

## Запрещено
- Пушить в `main` и `develop` напрямую.
- Мержить PR без одобрения.
- Оставлять закомментированный код.
- Мержить с падающими тестами.
